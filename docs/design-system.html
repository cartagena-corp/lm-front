The design system implemented in La Muralla

A rendered reference for every color token, type scale, spacing value, material, component recipe, and icon mapping shipped in this app — read this before touching any presentation-layer code in a new session. Full narrative version: docs/DESIGN_SYSTEM.md.

  1. 1Light mode only. Dark mode was deliberately removed — no .dark, no theme toggle, no data-theme branching anywhere.
  2. 2Sidebar is permanently #101828. Not a themeable surface — a fixed brand element.
  3. 3No floating notification widgets. Notifications live in a dropdown anchored inside the Topbar.
  4. 4Zero raw Tailwind palette classes. Every color is a CSS variable — a Geist scale step or a semantic alias.
  5. 5lucide-react only. The hand-rolled icon set and SVG sprite are deleted — don't reintroduce either pattern.

01 Color

All tokens live in src/assets/geist/colors.css, consumed directly as CSS variables — never through a Tailwind theme extension. Click any swatch to copy its variable name. Step convention: 100 bg tint · 200 hover bg · 300 active bg · 400 border · 500 hover border · 600 active border · 700 solid fill · 800 fill hover · 900 secondary text · 1000 primary text.

Brand primary --primary-*

The app's own scale, anchored on the brand navy #101828. This is the single unified recipe for "the one primary CTA button in a view" — it replaced four previously-competing recipes (black invert, ad-hoc blue+white, a text-white variant, and a purple-for-edit split).

Sidebar surface --sidebar-*

A deliberately separate namespace from --primary-* — the sidebar is a persistent surface, not a CTA, so its hover/active tuning must never cross-affect button tuning.

Only two allowed non-token colors in the whole app: ThemeScript.tsx's accent-bootstrap script (sets a runtime CSS variable's value, not a hardcoded UI color) and a "Blanco" #ffffff swatch inside TextArea.tsx's rich-text color picker (user content color, not UI chrome). Backend-driven status/priority/type chip colors are also intentionally left inline/data-driven — that's an established pattern, not a leftover.

02 Typography

Geist Sans for all UI and prose, Geist Mono for code/data/technical labels (loaded via next/font/google — this reference falls back to the same system stack the tokens define, since webfonts can't load here). Three weights only — 400 read, 500 interact, 600 announce — hierarchy comes from size + tracking, not weight.

03 Spacing & layout

4px base grid, --space-1 (4px) through --space-24 (96px, not all shown). Control heights: 32 / 40 / 48px for small / medium / large. Breakpoints are stock Tailwind — md (768px) is the sidebar-rail/mobile-drawer cutover, lg (1024px) is where side panels become rows.

Scroll-container rule: a flex-column ancestor chain needs min-h-0 at every level, not just the scrolling element — or a descendant's overflow-y-auto silently does nothing. This was the single most common layout bug found across the app.

04 Materials

Border-first. Static elements are defined by a 1px border or a zero-offset "shadow-as-border," never a soft drop-shadow. Real elevation shadow is reserved for hover states and things lifted above the content plane. Don't mix rounded and sharp corners in one view.

05 Component recipes

No component library exists — these are hand-repeated conventions across ~150 files. Match the recipe for a component's category exactly; don't improvise a new one.

Buttons

🗑 danger-hover accent

Primary → --primary-700/800 + --primary-contrast-fg. Danger → --red-700/800 + --ds-contrast-inverse. Secondary → --ds-card + shadow-border. Focus rings stay --blue-700 everywhere, even on primary buttons — the navy scale is reserved for CTA fills only.

Gotcha: a hover color must be a class (hover:bg-[var(--x)]), never inline style — inline styles always beat :hover specificity, so the hover silently never fires.

Card

A card is --ds-card background + shadow-border + radius-md. Hover this card.

06 Icon system

All icons are lucide-react (v1.23.0). The old hand-rolled icon set and SVG sprite are deleted — don't reintroduce either.

The stroke-width gotcha

Lucide takes strokeWidth, not stroke (that prop controls color, not weight, on a lucide icon). This app's characteristic weight is strokeWidth={1.5} — lucide's own default is 2, visibly heavier. Always pass it explicitly.

08 Known quirks worth knowing about

  • Hydration console-noise filter — HydrationNoiseFilter.tsx monkey-patches console.error in dev only to swallow browser-extension DOM-injection hydration warnings (e.g. bis_skin_checked). Deliberate, user-requested — not hiding a real bug.
  • No browser-automation tool was available during the redesign session — verification relied on npm run build + static reasoning. A manual visual QA pass is still owed, especially icon stroke-weight and primary-button contrast.
